Ember Addon Strategies and Best Practices

Anatomy of an Addon

  • Include Ember classes and make them available to parent app
  • Include new assets and make them part of the build pipeline
  • Use blueprints to generate new files

When to Addonize?

  • When you need assets from your parent
  • When you think assets you are creating would be well served elsewhere

Our Strategy

  • ember-purecloud-components to be the main addon designed for engine usage
  • Treating it as Interal Open Source (Core Team, Collaborators)
  • PR components, assets you need to the repo
  • go off reservation to Public OSS via

Broccoli from 10,000 Feet

Tips to Componentizing in addons

  • Push up services in favor of data binding/closure actions
  • Keep styles local to components, leverage common css variables (coming soon)
  • use the default blueprint for bower dependencies
  • use the included hook in index.js to import non-standard files to ember
  • put public files in /public, access them in parent app with via the addon name
  • if you insert into the pipleline (using broccoli hooks), you can view generated and intermediate data in tmp once the server is started. Also See Broccoli Stew}
